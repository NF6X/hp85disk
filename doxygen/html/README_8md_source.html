75 ## Makefile Configuration options for building
76  * Update BAUD, PORT, BOARD, PPR_REVERSE_BITS and RTC_SUPPORT for your platform
77  * BOARD is the version of the hardware
78  * 1 = V1 hardware without GPIB BUS transceivers
79  * 2 = V2 hardware with GPIB BUS transceivers
80  * PPR_REVERSE_BITS
81  * 0 = V1 hardware without the GPIB buffers
82  * RTC_SUPPORT for Real Time Clock
83  * 1 = RTC support for a DS1307 command compatible RTC chip - the DS3231 is the 3.3V version
84  * This will time stamp plot files and add time stamps inside lif images
85  * PORT is the emulator serial PORT name as detected by your operating system
86  * /dev/ttyUSB0 on my system
87  * BAUD is the data rate when connecting to the hp85disk emulator with a serial port
88  * 115200 = a safe default that most systems can manage
89  * NOTE: My development environment works with 500000 baud but 115200 should work on all systems
90  * NOTE: Faster is better when enabling more debug messages
91  * Too many messages can cause the HP85 to timeout waiting for IO
92  * AVRDUDE_DEVICE is the name of AVR as it is known by avrdude
93  * m1284p
94  * AVRDUDE_SPEED is the programming clock speed used by avrdude
95  * 5

97  * atmelice_isp

294 ## Important notes about SD Card requirments for the emulator
295  * Must be formatted FAT32
296  * The HP85 is sensitive to long read/write delays that some cards can cause problems with
297  * You want SD Cards with fast random writes
298  * I have found that the Sandisk Extreme and Sandisk Extreme Pro cards work best.
299  * There is a huge difference in various cards on the market.
300  * Look for the cards with the best 4K random write times
301  * A good source of benchmark information is looking for recent Raspberry Pi SD card benchmarks because they use SD Cards
302  * Specifically look at best 4k random write - faster is better.
303  * Why slow SD Cards are a problem?
304  * Summary: When the hp85disk emulator writes to the SD Card the Card internally must modify much much larger internal flash page - this can take too long
305  * Details:
306  * First step: hp85disk emulator writes to the SD Card
307  * Internally the SD card finds an internal page where our data will go
308  * Next the SD Card reads the page into internal RAM (recall it can be much over a megabyte)
309  * Next the SD Card modifies the internal RAM with our data
310  * Next the SD Card erases the page
311  * Lastly the page is written to the SD Card and this takes time
312  * Trivia - SD cards must to erase a page before updating because of the flash memory cell design
313  * Why is this delay critical to the hp85disk emulator?
314  * Too long of a delay can cause a timeout when writing to disk
315  * SD cards internal hardware is mostly optimized for sequential writing
316  * Their buffers and timing are designed primarily for writing consecutive blocks one after another
317  * When writing in consecutive order they can queue up many requests and combine them into one operation - a huge savings in time - done all in hardware
318  * Writing to blocks in random locations break this optimization very badly
319  * Therefore some SD Cards can take so long the HP85 can timeout
320  * The hp85disk emulator does not have enough memory to work around this issue
321  * If we could load the entire disk image into ram AND we had more for many write buffers
322  * Then we could optimize the SD Card writing process to avoid the problems
323  * The AVR we use has only 20K of ram for everything
324  * Perhaps some day we will port this project to a Raspberry PI with add on hardware
325  * The protocol is already solved for the emulator so this would not be that hard - only a battle with the time I have free

492 ## Understanding Drive GPIB BUS addressing and Parallel Poll Response (PPR) - HP85A vs. HP85B
493  * While GPIB devices can have address between 0 and 31 you can have no more than 8 disk drives.
494  * ALL disk drives are required to respond to a PPR query by the (HP85) controller.
495  * PPR query is done when the controller in charge (HP85) pulls ATN and EOI low.
496  * PPR response occurs when a disk drive pulls one GPIB bus data line low in response.
497  * You can only have 8 of these because there are only 8 GPIB data bus lines.
498  * GPIB data bus bits are numbered from 1 to 8
499  * PPR response bits are *assigned in reverse order* starting from 8, bit 8 for device 0
500  * This is a GPIB specification - not my idea.
501  * The HP85 uses these assumptions
502  * PPR bits are assigned in reverse order from device numbers.
503  * IMPORTANT! On power up the HP85 issues a PPR query for disk drives
504  * The emulator must be running BEFORE this happens.
505  * PPR query = both ATN and EOI being pulled low by the computer.
506  * PPR response is when each drive pulls a single GPIB data bus bit LOW - while ATN and EOI are low.
507  * *ONLY* those that are detected in this way are then next scanned
508  * Next for all detected drives the HP85 issues "Request Identify" to each in turn.
509  * This is done one drive at a time in order
510  * The PPR keyword in the [hpdisk.cfg](sdcard/hpdisk.cfg) is the PPR bit the drive uses
511  * PPR of 0 = PPR response on GPIB data bus bit number 8 - as per GPIB BUS specifications.
512  * The ID keyword in [hpdisk.cfg](sdcard/hpdisk.cfg) is the 16 bit reply to "Request Identify Reply"
513  * IMPORTANT! AMIGO drives cannot be queried for detailed drive layout information
514  * The HP85A can only use its *hard-coded firmware tables* to map ID to disk layout parameters
515  * This implies that the HP85A can only use AMIGO disks it has defined in firmware.
516  * The HP85B can query newer SS80 drives for detailed drive layout information instead.
517  * The HP85A cannot use SS80 drives unless it uses copies of the HP85B EMS and EDISK ROMS.
518  * One way this can be done with the PRM-85 expansion board offered by Bill Kotaska
519  * (The PRM-85 is great product giving you access to all of the useful ROMS)
520 ___

522 ## Technical Limitations
523  * Multiple drive support is implements but UNIT support is NOT
524  * While most AMIGO and SS80 feature have been implemented my primary focus was on the HP85A and HP85B.
525  * (I do not have access to other computers to test for full compatibility)
526  * This means that a few AMIGO and SS80 GPIB commands are not yet implemented!
527  * Some of these are extended reporting modes - many of which are optional.
528  * Note: The HP85A can only use AMIGO drives - unless you have the HP85B EMS ROM installed in your HPH9A
529  * This can be done with the PRM-85 expansion board offered by Bill Kotaska (a great product!)
530  * To attach a drive to our computer, real or otherwise, you must know:
531  * The correct GPIB BUS address and parallel pool response (PPR) bit number your computer expects.
532  * See ADDRESS, PPR and ID values in [hpdisk.cfg](hpdisk.cfg)
533  * Older computers may only support AMIGO drives.
534  * Such computers will have a hard coded in firmware list of drive its supports.
535  * These computers will issue a GPIB BUS "request identify" command and only detect those it knows about.
536  * *If these assumptions do NOT match the layout defined in the [hpdisk.cfg](sdcard/hpdisk.cfg) no drives will be detected.*
537  * Newer computers with SS80 support can request fully detailed disk layout instead of the "request identify"
538  * My emulator supports both reporting methods - but your computer may not use them both!
539  * For supported values consult your computer manuals or corresponding drive manual for your computer.
540  * See gpib/drives_parameters.txt for a list on some known value (CREDITS; these are from the HPDir project)
541  * In all cases the [hpdisk.cfg](sdcard/hpdisk.cfg) parameters MUST match these expectations.
542  * The [hpdisk.cfg](sdcard/hpdisk.cfg) file tells the emulator how the emulated disk is defined.
543  * GPIB BUS address, Parallel Poll Response bit number and AMIGO Request Identify response values.
544  * Additional detail for SS80 drives that newer computers can use.
545  * In ALL cases the file informs the code what parameters to emulate and report.
546  * ALL of these values MUST match your computers expectations for drives it knows about.
547  * Debugging
548  * You can enable reporting of all unimplemented GPIB commands (see *TODO* debug option in [hpdisk.cfg](sdcard/hpdisk.cfg) )
549  * Useful if you are trying this on a non HP85 device
550  * See the [hpdisk.cfg](sdcard/hpdisk.cfg) for documentation on the full list of debugging options
551  * The emulator can passively log all transactions between real hardware on the GPIB bus
552  * Use the "gpib trace *logfile*" command - pressing any key exits - no emulation is done in this mode.
553  * You can use this to help understand what is sent to and from your real disks.
554  * I use this feature to help prioritize which commands I first implemented.
