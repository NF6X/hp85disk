39 ## Features
40  * This project emulates GPIB drives and HPGL printer for the HP85A and HP85B computers.
41  * Each drive can be fully defined in the hpdisk.cfg file on the SD CARD
42  * AMIGO drives work with HP85A
43  * SS80 drives work with HP85B (or HP85A with prm-85 add on board see links)
44  * Printer emulator - can capture and save printer data to a time stamped file.
45  * You can connect with and control the emulator via its FTI USB serial interface 115200 baud, 8N1.
46  * There are many commands that you can use, type "help for a list"
47  * Any key press halts the emulator and waits for a user command.
48  * After finishing any user commend it returns to GPIB disk emulation.
49  * Each emulated disk image is a single file on a FAT32 formatted SDCARD.
50  * Internally these disk images are formatted using the LIF standard.
51  * LIF format used is compatible with HP85A/B and many other computers
52  * LIF Tools are built into the emulator to create, rename, delete add and extract to/from other LIF images.
53  * LIF tools can also create new LIF images with user specifications
54  * The emulator will automatically create missing LIF images defined and named in hpdisk.cfg on the SDCARD
55  * For the specific LIF E010..E013(hex) type images the emulator can translate to and from plain ASCII files.
56  * The emulator can add and extract as LIF image format
57  * You may add a single file from another LIF image with multiple internal files.
58  * You may extract a single file from a LIF image to a new LIF image.
59  * Extracted images have a 256 byte volume header, 256 byte directory followed by a file.
60  * Type "lif help" in the emulator for a full list of commands
61  * See the top of [lifutils.c](lif/lifutils.c) for full documentation and examples.
62  * TeleDisk to LIF extractor tool included - see [lif README.md](lif/README.md)
63  * [td02lif](lif/t202lif) [85-SS80.TD0](lif/85-SS80.TD0) [85-SS80.LIF](lif/85-SS80.LIF)

66 ## SD Card requirments
67  * The HP85 is sensitive to write delays so we need SD cards with fast random writes.
68  * I have found that the Sandisk Extreme and Sandisk Extreme Pro cards work best.
69  * Why ? Each block that is written reads, erases then rewrites a new internal flash page (these can be over a megabyte and so take time). Most SD cards are optimized for sequencial writting and do not do well with random writes. There is a huge difference in various cards on the market. Look for the cards with the best 4K random write times. Some SD cards are so slow it will cause the HP85 to timeout waiting for the card. Best source of benchmark information is looking for recent Raspberry Pi SD card benchmarks - specifically 4k random write - faster is better.

129 ## Understanding Drive GPIB BUS addressing and Parallel Poll Response (PPR) - HP85A vs. HP85B
130  * While GPIB devices can have address between 0 and 31 you can have no more than 8 disk drives.
131  * ALL disk drives are required to respond to a PPR query by the (HP85) controller.
132  * PPR query is done when the controller in charge (HP85) pulls ATN and EOI low.
133  * PPR response occurs when a disk drive pulls one GPIB bus data line low in response.
134  * You can only have 8 of these because there are only 8 GPIB data bus lines.
135  * GPIB data bus bits are numbered from 1 to 8
136  * PPR response bits are *assigned in reverse order* starting from 8, bit 8 for device 0
137  * This is a GPIB specification - not my idea.
138  * The HP85 uses these assumptions
139  * PPR bits are assigned in reverse order from device numbers.
140  * IMPORTANT! On power up the HP85 issues a PPR query for disk drives
141  * The emulator must be running BEFORE this happens.
142  * PPR query = both ATN and EOI being pulled low by the computer.
143  * PPR response is when each drive pulls a single GPIB data bus bit LOW - while ATN and EOI are low.
144  * *ONLY* those that are detected in this way are then next scanned
145  * Next for all detected drives the HP85 issues "Request Identify" to each in turn.
146  * This is done one drive at a time in order
147  * The PPR keyword in the [hpdisk.cfg](sdcard/hpdisk.cfg) is the PPR bit the drive uses
148  * PPR of 0 = PPR response on GPIB data bus bit number 8 - as per GPIB BUS specifications.
149  * The ID keyword in [hpdisk.cfg](sdcard/hpdisk.cfg) is the 16 bit reply to "Request Identify Reply"
150  * IMPORTANT! AMIGO drives cannot be queried for detailed drive layout information
151  * The HP85A can only use its *hardcoded firmware tables* to map ID to disk layout parameters
152  * This implies that the HP85A can only use AMIGO disks it has defined in firmware.
153  * The HP85B can query newer SS80 drives for detailed drive layout information instead.
154  * The HP85A cannot use SS80 drives unless it uses copies of the HP85B EMS and EDISK ROMS.
155  * One way this can be done with the PRM-85 expansion board offered by Bill Kotaska
156  * (The PRM-85 is great product giving you access to all of the useful ROMS)

158 ## Limitations
159  * Multiple drive support is impliments but UNIT support is NOT
160  * While most AMIGO and SS80 feature have been implemented my primary focus was on the HP85A and HP85B.
161  * (I do not have access to other computers to test for full compatibility)
162  * This means that a few AMIGO and SS80 GPIB commands are not yet implemented!
163  * Some of these are extended reporting modes - many of which are optional.
164  * Note: The HP85A can only use AMIGO drives - unless you have the HP85B EMS ROM installed in your HPH9A
165  * This can be done with the PRM-85 expansion board offered by Bill Kotaska (a great product!)
166  * To attach a drive to our computer, real or otherwise, you must know:
167  * The correct GPIB BUS address and parallel pool response (PPR) bit number your computer expects.
168  * See ADDRESS, PPR and ID values in [hpdisk.cfg](sdcard.cfg)
169  * Older computers may only support AMIGO drives.
170  * Such computers will have a hard coded in firmware list of drive its supports.
171  * These computers will issue a GPIB BUS "request identify" command and only detect those it knows about.
172  * *If these assumptions do NOT match the layout defined in the [hpdisk.cfg](sdcard/sdcard.cfg) no drives will be detected.*
173  * Newer computers with SS80 support can request fully detailed disk layout instead of the "request identify"
174  * My emulator supports both reporting methods - but your computer may not use them both!
175  * For supported values consult your computer manuals or corresponding drive manual for your computer.
176  * See gpib/drives_parameters.txt for a list on some known value (CREDITS; these are from the HPDir project)
177  * In all cases the [hpdisk.cfg](sdcard/hpdisk.cfg) parameters MUST match these expectations.
178  * The [hpdisk.cfg](sdcard/hpdisk.cfg) file tells the emulator how the emulated disk is defined.
179  * GPIB BUS address, Parallel Poll Response bit number and AMIGO Request Identify response values.
180  * Additional detail for SS80 drives that newer computers can use.
181  * In ALL cases the file informs the code what parameters to emulate and report.
182  * ALL of these values MUST match your computers expectations for drives it knows about.
183  * Debugging
184  * You can enable reporting of all unimplemented GPIB commands (see *TODO* debug option in [hpdisk.cfg](sdcard/hpdisk.cfg) )
185  * Useful if you are trying this on a non HP85 device
186  * See the [hpdisk.cfg](sdcard/hpdisk.cfg) for documentation on the full list of debugging options
187  * The emulator can passively log all transactions between real hardware on the GPIB bus
188  * Use the "gpib trace *logfile*" command - pressing any key exits - no emulation is done in this mode.
189  * You can use this to help understand what is sent to and from your real disks.
190  * I use this feature to help prioritize which commands I first implemented.
