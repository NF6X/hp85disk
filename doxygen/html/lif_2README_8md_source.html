145 ## LIF command summary
146  * Built in help
147  * **lif help**
148  * Gives lif commands
149  * Summary - List, Create, import, export, translate, copy, rename, delete, etc
150  * NOTE: Each disk image is a single file, encoded in **LIF** format,saved on the SD Card
151  * **LIF** format is a common the filesystem on series 80 computers.
152  * **LIF** format is also a vary common file interchange format for series 80 computers
153  * **LIF** format includes file date,size permissions and other important meta data
154  * Translate between DTA8x (type E010) and plain text files
155  * You can add a plain text file, and translate it, into a **LIF** image with file format DTA8x (type E010)
156  * You can extract and translate DTA8x (type E010) into a plain text files
157  * **Key LIF manipulation features**
158  * **dir** display a directory of a **LIF** image
159  * Directory listing of **LIF** images and SSD Card files
160  * If you have an RTC the listing can display file and **LIF** volume date and time
161  * Display time stamps if they were set
162  * But only if they were created or added with the built in tools
163  * **add** an ascii file to **LIF** image
164  * This function permits renaming of the translated file
165  * They get translated between **HP85** DTA8x (type E010) format and plain text files!!!
166  * **extract** ASCII files from **LIF** image
167  * This function permits renaming of the translated file
168  * They get translated between **HP85** DTA8x (type E010) format and plain text files!!!
169  * **addbin** add binary programs from one **LIF** image to another *LIF* image
170  * This function permits renaming of the translated file
171  * **extractbin** a single binary file or program into a new **LIF** image
172  * This function permits renaming of the translated file
173  * Extracted **LIF** images contain a single file a 256 byte volume header, 256 byte directory followed by a file.
174  * **del** delete file in **LIF** image
175  * **rename** file in **LIF** image
176  * **create** create a LIF image specifing a label, directory size and overall disk size
177  * **createdisk** create a LIF image specifying a label and drive model name
178  * [For more **LIF** documentation](lif/README.md)

182 ## LIF command examples
183  * lif add lifimage lifname from_ascii_file
184  * Add ASCII file converted to E010 format to existing LIF image on SD card
185  * lif add /amigo1.lif TEST1 /test.bas
186  * lif add /amigo1.lif TREK85 /TREK85/TREK85.BAS
187  * Notes:
188  * Strings must be no longer then sector size - 3
189  * Any trailing "\n" and/or "\r" are coverted to "\n" when stored in LIF file
190 
191  * lif addbin lifimage lifname from_lif_image
192  * Adding a binary LIF image files to another LIF image
193  * Examples
194  * lif addbin /amigo1.lif TEST /test.lif
195  * lif addbin /amigo1.lif TREK85 /TREK85/trek.lif
196  * Notes about TREK85 in the examples
197  * Author: TREK85 port was done by Martin Hepperle
198  * https://groups.io/g/hpseries80/topic/star_trek_game_for_hp_85/4845241
199 
200  * lif create lifimage liflabel directory_sectors total_sectors
201  * Create a new LIF image
202  * Example
203  * lif create /amigo3.lif AMIGO3 15 1120
204  * This formats an LIF image file with 15 directory sectors and a length of 1120 (16 * 2 * 35) sectors
205 
206  * lif del lifimage lifname

208  * Example

279 ## TELEDISK conversion technical notes - extracting LIF data
280  * Unfortunately TeleDisk saves everything in on all sides and tracks.
281  * Question: what happens when we have a disk formatted multiple times with differing parameters ?
282 
283  * Example 1
284  * Say we start an image that was two sided 80 track disk, 9 sectors, 512 byte sectors
285  * Later on someone formats it single sides 35 track disk, 16 sectors, 256 bytes sectors
286  * Analisis
287  * We examing the first 30 tracks on both sides looking for:
288  * FIRST sector on both sides and its size
289  * LAST sector on both size that matches the size
290  * COUNT of sectors on each side that matches the size
291  * We then compare sector COUNT, compare sector SIZE, and FIRSTd
292  * If SIZE or COUNT is not the same reject
293  * If only FISRT mismatches we then check FIRST for continuation of numbering on each side
294  * Conclusion
295  * We can reject side two by comparing the
296  * FIRST sector number, Number of sectors and size
297 
298  * Eample 2
299  * We have a double sided disk that is formated single sided with FIRST,LAST,COUNT and SIZE matching
300  * Analisis
301  * We have no automated way to reject side two
302  * Conclusion
303  * The user will have to override heads

308 ## TELEDISK extracting LIF data user overrides
309  * Number of Sides:
310  * You CAN specify single sided override for two sided disks
311  * (as defined in the TeleDisk headers)
312  * However you can NOT specify 2 sided mode if the TeleDisk image headers say there is one side
313 
314  * Sector Size override
315  * Rather then using the SIZE of the FIRST sector as a filter the user can manually pick the size
316 
317  * Number of tracks overrides
318  * The LIF decoder stops after the last sector of the last file so this option is ignored
319 </pre>
